## 三、v1.0+ 验证期（0-3个月）

### 3.1 核心目标

**不是开发新功能，而是让现有功能被更多人知道和使用。**

### 3.2 关键行动

#### 行动 1：降低首次体验门槛（P0）

**问题**：v1.0.0 功能完整，但新用户可能觉得"太重"

**解决方案**：

```bash
# 新增 --quick 模式（不是 --lite，而是快速体验）
harness-init my-project --quick

# 生成的简化体验项目
my-project/
├── src/
├── tests/
├── README.md
├── pyproject.toml
└── Makefile

# 不包含（但可以随时添加）：
# - .harness/ 完整工作区
# - AGENTS.md 完整规范
# - 复杂文档体系
```

**价值**：
- 5 分钟跑通，体验核心验证流程
- 满意后再启用完整 Harness 工作流
- 渐进式采用，降低心理门槛

**时间**：1 周  
**版本**：v1.0.1

---

#### 行动 2：跨平台验证命令（P0）

**问题**：Windows 用户没有 make，首次体验失败

**解决方案**：

```python
# 新增 harness 命令组
harness verify      # 跨平台验证，替代 make verify
harness test        # 运行测试
harness lint        # 代码检查
harness init        # 重新初始化（已有功能）
```

**实现**：
```python
# src/harness_init/commands.py
def verify_command():
    """跨平台验证命令."""
    import subprocess
    import sys
    
    # 自动检测平台
    if sys.platform == "win32":
        # Windows: 直接调用 Python 脚本
        run_ruff_windows()
        run_pytest_windows()
    else:
        # Unix: 优先使用 make，否则 Python 脚本
        if has_make():
            subprocess.run(["make", "verify"])
        else:
            run_verify_python()
```

**时间**：1 周  
**版本**：v1.0.2

## 八、总结

### 核心认知转变

```
从 "功能开发" 转向 "市场验证"
从 "我要做什么" 转向 "用户需要什么"
从 "技术完美" 转向 "有人使用"
```

### v1.0+ 的核心理念

> **功能已经完善，现在需要证明有人需要它。**

v1.0.0 是一个**完整的产品**，不是 MVP。
现在的任务是：
1. 让更多人知道它
2. 让尝试的人能快速成功
3. 收集反馈决定下一步

- 100+ stars
