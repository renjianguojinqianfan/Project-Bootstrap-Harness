\# PBH 设计文档（DESIGN.md）



\## 1. 核心设计哲学



\*\*三种叙事\*\*：

\- 本质："我们不干预 AI 的思考，我们约束它发生的环境。"

\- 形式："它看起来像 .editorconfig，但作用更接近 CI。"

\- 价值："PBH 存在的意义，是让每一次重新向 AI 解释项目规则，变得不可接受。"



\*\*PBH 三问（所有设计决策的自检标准）\*\*：

1\. 这是不是在干预 AI 思考？→ 是，就拒绝。

2\. 这是不是在降低"没有 PBH 的痛苦"？→ 是，就在削弱自己。

3\. 这是不是让协议更难被忽略？→ 否，就没价值。



\## 2. 架构分层



PBH 核心只做一层：\*\*Protocol Seeder（协议播种）\*\*。

\- 入口层 (cli)：参数解析，仅做路由

\- 引擎层 (core)：模板渲染 + 参数化生成

\- 模板层 (templates)：协议文件 + 质量门禁



PBH 核心不包含、也不会包含：

\- Agent 运行时

\- 代码生成

\- 行为监控



让协议"动起来"是生态工具（Harness-Lint、Harness-Test 等）的职责。



\## 3. 关键设计决策



\### 3.1 为什么砍掉 v1.0 的 Agent 运行时代码？

因为那些代码在"干预 AI 思考"，同时在"降低没有 PBH 的痛苦"。

它们让 PBH 看起来像在做运行时控制，但实际上什么都没做。

砍掉之后，PBH 的核心定位——"只播种，不运行"——才真正清晰。



\### 3.2 为什么选择 B 方案（柔性引导）而非 A 方案（强制角色）？

A 方案定义 Planner/Generator/Evaluator 三角色，这会绑定一种特定的 Agent 协作模式。

B 方案不定义角色，只给行为准则。这让 PBH 可以适配任何 Agent 框架，

同时让 PBH 自身成为一个观察不同 Agent 指令遵循程度的实验平台。



\### 3.3 为什么不让 PBH 定义机器可读的规则文件？

一旦定义 rules.yaml 之类的东西，PBH 就从"环境播种者"变成了"行为控制者"。

控制 AI 行为是 Agent 运行时的职责，不是 PBH 的职责。

PBH 的护城河恰恰在于它的不越界。



\## 4. 约束与防腐



\### 4.1 三层防腐机制

1\. \*\*架构分层隔离\*\*：核心只播种，任何让核心进入"执行"的提议即越界

2\. \*\*CI 心智锚定\*\*：make verify 是阻断性门禁，不是建议

3\. \*\*生态工具让失败可见\*\*：Harness-Lint 将协议遵守情况转化为可视化报告



\### 4.2 归因锚定原则

所有 PBH 生态工具的输出，必须可追溯到 AGENTS.md 的具体条款。

任何无法指向 AGENTS.md 的规则输出，都是不完整的。



\### 4.3 协议稳定性

\- progress.json 格式从 v1.5.0 起冻结

\- harness-version 标记确保生态工具可判断兼容性

\- AGENTS.md 采用机场导航式结构，章节编号预留扩展位
