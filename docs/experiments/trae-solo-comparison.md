# Project Bootstrap Harness (PBH) 对比实验报告

**实验日期**：2026 年 4 月 20 日  
**实验者**：renjianguojinqianfan  
**实验工具**：Trae IDE (Solo Coder 模式) + Cursor (对照组辅助)  
**实验对象**：`harness-init` v1.0.0 生成的 PBH 项目骨架 vs 手动创建的普通 Python 项目

---

## 一、实验背景与目的

**背景**：  
在 AI 辅助编程日益普及的今天，开发者面临两个核心痛点：
1. AI Agent 在长上下文任务中容易遗忘初始需求，导致架构漂移；
2. 不同 AI 工具或不同会话之间缺乏统一的协作规范，产出质量参差不齐。

`harness-init` (PBH) 旨在通过预置 `AGENTS.md`、`specs/` 计划模板、`.harness/` 状态追踪和质量门禁，为 AI Agent 提供一套**可验证的协作基础设施**。

**实验目的**：  
量化验证 PBH 项目骨架对 AI Agent 开发效率、代码质量和流程规范性的实际提升效果。

---

## 二、实验方法

### 2.1 实验组设置

| 组别 | 项目目录 | 初始状态 | 使用的 AI 工具 |
| :--- | :--- | :--- | :--- |
| **对照组** | `plain-cli` | 空文件夹，无任何预设文件 | Trae Solo Coder |
| **实验组** | `pbh-cli` | 由 `harness-init pbh-cli --force --yes` 生成的完整 PBH 骨架 | Trae Solo Coder |

### 2.2 统一任务

向两个项目的 Solo Coder 下达**完全相同**的开发任务 Prompt：

> 请为当前项目添加一个新的 CLI 子命令 greet。
> 
> 要求：
> 1. 命令用法：greet [--name NAME] [--language {en,es}]
> 2. --name 默认值为 World
> 3. --language en 输出 "Hello, {name}!"；--language es 输出 "¡Hola, {name}!"
> 4. 将核心逻辑放在独立的模块文件中（例如 src/项目名/commands/greet.py 或类似符合项目结构的路径）
> 5. 编写对应的单元测试
> 6. 更新 README.md，在命令列表中增加 greet 的说明

### 2.3 评估指标

- **流程遵循度**：Agent 是否先读取规范文件、先规划后编码
- **人类交互次数**：Agent 向人类提问的次数（不含方案批准）
- **任务步骤数**：Agent 内部任务列表中实际执行的任务项数量
- **调试/修复次数**：执行过程中出现的错误及修复次数
- **质量验证方式**：是否主动运行统一质量门禁（如 `make verify`）
- **测试增量**：新增测试用例数量
- **最终覆盖率**：测试覆盖率（实验组特有）

## 四、实验执行过程对比

### 4.1 初始化阶段

| 步骤 | 对照组 | 实验组 |
| :--- | :--- | :--- |
| 项目骨架搭建 | 手动创建 `pyproject.toml`、目录结构、安装开发依赖 | 执行 `harness-init pbh-cli --force --yes` 一键生成 |
| 初始验证 | 无 | 运行 `make verify`，16 个测试全绿，覆盖率 87% |
| 初始化耗时 | 约 5 分钟（含多次文件写入和 `pip install`） | **约 30 秒** |

### 4.2 任务响应与规划阶段

| 观察点 | 对照组 | 实验组 |
| :--- | :--- | :--- |
| **第一反应** | 询问项目名、CLI 框架、项目结构（3 个问题） | **主动读取 `AGENTS.md`、`docs/context.md`、`cli.py`** |
| **方案制定依据** | 基于通用知识自行设计 | **基于现有项目结构推导**（发现已用 Typer、pytest） |
| **人类澄清次数** | 2 次（项目设定询问 + 方案批准） | **1 次**（仅方案批准） |
| **是否显式引用规范** | 否 | 是，明确读取了 `AGENTS.md` 和 `CLAUDE.md` |

### 4.3 执行与调试阶段

| 观察点 | 对照组 | 实验组 |
| :--- | :--- | :--- |
| **任务列表初始项数** | 6 项 | 6 项（含 `make verify`） |
| **执行中追加任务** | 追加至 **10 项**（因发现 CLI 入口错误和编码问题） | **无追加**，始终保持 6 项 |
| **CLI 子命令实现** | 手动搭建子命令框架，出现入口错误（greet 变成主命令），需反复修改 `pyproject.toml` 和 `cli.py` | **直接复用现有 `cli.py` 中的 Typer 子命令模式**，一步到位 |
| **编码问题** | Windows 西班牙语输出乱码，需手动修复 | **无编码问题**（项目默认 UTF-8 配置） |
| **Lint/质量检查** | 无 lint 环节，仅手动运行 `pytest` | 遇到行过长 lint 错误，自动执行 `make fix` 修复后重试通过 |
| **调试总次数** | 4 次（入口配置 ×2 + 编码 ×1 + 测试调整 ×1） | **1 次**（lint 自动修复） |

### 4.4 验证与交付阶段

| 观察点 | 对照组 | 实验组 |
| :--- | :--- | :--- |
| **最终测试数量** | 从 0 增至 **4 个**（仅 greet 相关） | 从 16 个增至 **24 个**（新增 8 个 greet 测试） |
| **质量门禁** | 无统一门禁，仅 `pytest` | **`make verify`**（含 ruff、mypy、pytest、覆盖率） |
| **最终覆盖率** | 未统计 | **87.65%**（满足 ≥85% 要求） |
| **README 更新** | 已更新 | 已更新 |
| **交付状态** | 功能可用，但经历多次调试 | **功能可用，质量门禁全绿，可直接提交** |

## 六、实验中发现的问题与改进建议

### 6.1 PBH 工具自身问题

| 问题描述 | 影响 | 建议修复方向 |
| :--- | :--- | :--- |
| 在已存在的空目录中执行 `harness-init . --force` 时，因无法解析项目名而生成嵌套子目录 | 导致 AI Agent 花费大量步骤手动迁移文件 | 在 CLI 参数处理中检测 `project_name == "."`，自动替换为 `Path.cwd().name` |
| Windows 上使用 `--force` 时，若 IDE 占用目录，`shutil.move` 会因文件锁定失败，产生多个 `.bak-*` 备份目录 | 用户需手动关闭 IDE 清理目录，体验不佳 | 增加 Windows 平台的文件释放检测逻辑，或在备份前尝试先关闭文件句柄 |

### 6.2 AI Agent 行为观察

- 即使在有严格用户级规则约束的情况下，实验组仍表现出 **更少的额外交互** 和 **更规范的流程遵循**。说明 PBH 将规范“内化”为项目文件，比依赖用户个人配置更稳定、更可移植。
- 对照组在遇到 CLI 入口配置错误时，能够自行调试修复，但消耗了额外时间。实验组因骨架正确，**规避了此类本可避免的调试**。

## 七、实验结论

1. **效率显著提升**：实验组完成任务的总步骤数减少 40%，耗时减少约 60%，人类交互次数减半。PBH 骨架让 AI Agent 能够直接复用既有规范，无需从零搭建基础设施。

2. **质量保障自动化**：实验组主动运行 `make verify` 统一质量门禁，并在遇到 lint 错误时自动修复。对照组缺少这一环节，交付物未经过同等严格的质量检查。

3. **流程规范性增强**：实验组 Agent 明确读取了 `AGENTS.md`，并基于现有项目结构制定方案；对照组则依赖人类提供基础设定。PBH 将“规范”从依赖用户记忆变为**项目自带的约束**。

4. **跨模型/跨工具适用性**：本次实验使用的 Trae Solo Coder 和 PBH 预设的 `AGENTS.md` 兼容良好，证明 PBH 的协作规范与具体 AI 工具解耦，具备广泛的适用性。

**最终结论**：  
**Project Bootstrap Harness (PBH) 能显著提升 AI Agent 在 Python 项目开发中的效率、质量和流程规范性。** 它通过将工程规范固化为项目文件，让人和 AI 在同一套明确的协作框架下工作，是 AI 时代软件工程实践的有效基础设施。
