# PBH 改进计划 v2.0

> **定位**: PBH 是一个为 AI 编码时代设计的 **标准化项目结构生成器**。它只负责搭台子，不唱戏。
>
> **创建日期**: 2026-04-23
> **最后更新**: 2026-04-26
> **依据**: 笔记库分析 + 用户边界确认 + 智谱 Coding Agent 体系 / Hermes Profiles 架构 + 系统哲学对齐（三层防腐 / 归因锚定 / 不可绕过性）
> **状态**: 已定稿。Phase 1 已完成（v1.1.5）。Phase 2 待开发（v1.5.0）。Phase 3 规划中（v2.0.0）。

---

## 一、定位校准

### PBH 不是

- ❌ **不是 Agent 框架**（如 AutoGPT、CrewAI、Hermes Profiles）
- ❌ **不是 Spec Coding 流程工具**（不生成 PRD、SPEC、constraints.md）
- ❌ **不是代码生成器**（不替用户写业务逻辑）
- ❌ **不对 AI 角色进行定义**（不区分 Planner/Dev/QA 角色）

### PBH 是

- ✅ **一个预配置质量门禁的项目结构生成器**（`make verify` 开箱即用）
- ✅ **一个所有 AI 都能理解的标准化工作环境**（Cursor / Claude / Copilot / OpenCode / Trae）
- ✅ **一个为 Spec Coding 提供容器的脚手架**（创建 `tasks/` 目录，内容留给用户）
- ✅ **一个"协议的一次性播种器"**：在项目创建时种下规则，让约束随项目走，不随人走。

### 核心叙事

> **本质（What）**："我们不干预 AI 的思考，我们约束它发生的环境。"
>
> **形式（How it looks）**："它看起来像 `.editorconfig`，但作用更接近 CI。"
>
> **结果（Why it matters）**："PBH 存在的意义，是让每一次重新向 AI 解释项目规则，变得不可接受。"

## 二、改进方向

### 2.1 砍掉 Agent 运行时设施（P0）

**从标准模板中移除**:

| 移除项 | 原因 |
|--------|------|
| `src/{pkg}/harness/`（runner.py、evaluator.py、state.py、workflow.py）| Agent 运行时，非脚手架职责 |
| `src/{pkg}/agents/`（planner.py、generator.py、evaluator.py）| Agent 角色定义，非脚手架职责 |
| `src/{pkg}/tools/`（空 stub）| 无意义占位 |
| `.harness/plans/`、`eval_feedback/`、`state/` | Agent 状态管理，非脚手架职责 |
| `tests/test_harness.py` | 测试 Agent 运行时，一起移除 |

**保留**:
- `.harness/` 只保留 `templates/`（plan_template.json，供开发者自己参考）和 `progress.json`
- 这些是轻量级容器，不运行任何逻辑

---

### 2.2 AGENTS.md 重写为"机场导航"式（P0）

**目标**: 让刚接触项目的 AI 在 **30 秒内**安全开始工作。

**当前问题**: 宣言式结构（Principle → Protocol → Workflow → File Mapping），AI 读完还是不知道第一步该干嘛。

**目标结构**:

```markdown
# {project_name}

> 一句话描述项目要解决的问题。

## ⚡ 快速开始（30 秒上手）

1. `make verify` — 确认环境没问题
2. `pytest tests/` — 跑通现有测试
3. `python -m {package} --help` — 看程序怎么用

## ⚠️ 致命规则（The Law）

- **铁律**: 严禁在未通过 `make verify` 的情况下直接提交代码。
- **覆盖率**: 测试覆盖率不得低于 85%，低于会 CI 失败。
- **代码风格**: 请模仿 `src/{package}/utils.py` 的写法。

## ✍️ 行为准则

- 你是一位资深软件工程师。
- 写代码前，先说清楚你要改什么、为什么这样改。
- 所有公开 API 必须有完整的 type hints 和 docstring。

## 📂 目录速查

| 位置 | 用途 |
|------|------|
| `src/{package}/` | 主代码 |
| `tests/` | 测试（镜像 src/ 结构）|
| `docs/` | 文档 |
| `tasks/` | 任务拆解（Spec Coding 容器）|

## 🧰 常用命令

| 命令 | 作用 |
|------|------|
| `make verify` | lint + test + coverage（必须过） |
| `make test` | 只跑测试 |
| `make lint` | 只跑 ruff |
| `make fix` | 自动修复 lint 问题 |
```

**Quick 模式版本**（≤50 行，保留"致命规则"和"快速开始"）：

```markdown
# {project_name}

> 一句话描述。

## ⚡ 快速开始
1. `make verify`
2. `pytest tests/`

## ⚠️ 致命规则
- 必须通过 `make verify` 才能提交
- 覆盖率 ≥85%

## 📂 目录
`src/{pkg}/` 主代码 | `tests/` 测试 | `tasks/` 任务拆解
```

### 2.3 IDE 按需配置（P1）

**新增 `--ide` 参数**，避免项目里躺一堆用不上的文件：

```bash
harness-init my-project --ide=cursor      # 只生成 .cursorrules
harness-init my-project --ide=claude      # 只生成 CLAUDE.md
harness-init my-project --ide=trae       # 生成 .trae/rules/
harness-init my-project --ide=copilot    # 生成 .github/copilot-instructions.md
harness-init my-project --ide=opencode    # 生成 opencode.yaml
harness-init my-project --ide=all        # 全部（默认）
```

**实现**: `core.py` `_should_copy()` 增加 ide 过滤逻辑。

---

### 2.4 项目类型扩展（P1）

**新增 `--template` 参数**，支持多种项目起点：

| 类型 | 入口 | 核心依赖 | 场景 |
|------|------|---------|------|
| `cli`（默认）| `cli.py` | typer, rich | 命令行工具 |
| `lib` | `__init__.py` + API | 无 CLI 依赖 | 可复用库 |
| `web` | `main.py` + FastAPI | fastapi, uvicorn | API 服务 |
| `notebook` | `notebooks/` | jupyter, pandas | 数据分析 |

**实现**: `templates/` 下按类型分子目录，共享文件放 `templates/common/`。

---

### 2.5 Quick 模式校准（P2）

**核心原则**: 约束文件是必需品，不是奢侈品。Quick 模式保留行为约束，砍掉重型设施。

**Quick 模式保留**:
- `.cursorrules` 或 `CLAUDE.md`（根据 `--ide`）
- `AGENTS.md`（精简版，≤50 行，保留"致命规则"和"快速开始"）
- `Makefile`（`verify`/`test`/`lint`）
- `pyproject.toml`（最小依赖：只有 typer + pytest + ruff，去掉 pydantic/pyyaml/rich）
- `README.md`（精简版）
- `tests/` + `test_cli.py`（保持测试习惯）
- `tasks/`（空容器）

**Quick 模式排除**:
- `.github/`（CI，先本地验证）
- `docs/decisions/`（ADR，太重）
- `scripts/`（pre-push hooks）
- `configs/`（环境配置）
- `harness/` / `agents/` / `tools/` / `utils/` stubs（已砍掉）
- `README.en.md`（国际化，Quick 不需要）

---

### 2.6 `tasks/` 目录：Spec Coding 容器（P2）

**设计**: PBH 核心只创建目录和空模板，任务生成和管理交给用户或未来独立工具。

```
tasks/
├── task_plan.md          # 带注释引导的空模板
└── .gitkeep              # 保持目录在 git 中
```

`task_plan.md` 模板内容:

```markdown
# Task Plan

> 使用 Spec Coding 流程，按 Phase 拆解任务。
> 每个 Task 独立交付、独立测试、独立回滚。

## Phase 1: 基础设施（无依赖）
- [ ] T1: ... - 验收: ...

## Phase 2: 核心模块
- [ ] T2: ... - 依赖 T1 - 验收: ...
```

**不做的事**: 不生成 PRD.md、SPEC.md、constraints.md。这些是 Spec Coding 流程的产物，由开发者自己写。

---

### 2.7 模板自检命令（P3）

**新增**:

```bash
harness-init validate     # 生成临时项目，跑 make verify，确认模板可用
harness-init doctor       # 检查本地环境（Python 版本、make、ruff、git）
```

**用途**:
- 用户安装后跑 `doctor` 确认环境就绪
- PBH 自身 CI 跑 `validate` 确保发布前模板可用
- 解决当前 `test_generated_project_passes_make_verify` 因环境失败的问题

## 三、不做的事

| 不做 | 原因 |
|------|------|
| 不生成 PRD.md / SPEC.md / constraints.md | Spec Coding 流程产物，开发者自己写 |
| 不设计 Agent 运行时 | 不是 PBH 定位 |
| 不实现 Harness Engine | 不是脚手架该管的 |
| 不生成代码规范长文档 | `.cursorrules` / `CLAUDE.md` 已承担约束角色 |
| 不定义 AI 角色（Planner/Dev/QA）| 边界外 |
| 不定义机器可读规则文件（rules.yaml 等） | 控制 AI 行为是 Agent 运行时的职责，非 PBH 本分 |
| 不让 PBH 核心进入"执行"或"监控"状态 | 只播种，不运行。运行是生态工具的事 |

## 四、实施计划

### Phase 1: 定位校准（v1.1）
**目标**: 砍掉越界功能，重写核心模板，确立 PBH 的清晰边界。

| 任务 | 文件 | 工作量 |
|------|------|--------|
| 砍掉 harness/agents/tools stubs | `core.py` 移除模板引用，`templates/` 删除目录 | 小 |
| 重写 AGENTS.md 模板 | `templates/AGENTS.md`、`templates/AGENTS.quick.md` | 中 |
| 重写 Quick 模式排除列表 | `core.py` `_QUICK_MODE_EXCLUSIONS` | 小 |
| 调整 pyproject.toml Quick 版 | `templates/pyproject.quick.toml` 精简依赖 | 小 |
| 更新测试 | `tests/test_core.py`、`tests/test_cli.py` 移除 harness/agents 相关断言 | 中 |

**验证**:
```bash
pytest tests/ -q            # 全部通过
harness-init validate       # 模板自检通过
make verify                 # PBH 自身质量通过
```

**Phase 1 实地验证（用 OpenCode + OMO）**：

```bash
# 1. 用新 PBH 初始化测试项目
harness-init test-project --ide=opencode

# 2. 进入项目，跑 AI 协作测试
cd test-project
# 用 OpenCode 打开，验证 AI 是否能在 30 秒内理解项目并开始工作
# 检查点：AI 能否正确调用 `make verify`、找到入口文件、理解目录结构

# 3. 跑质量门禁
make verify
```

**验证标准**：OpenCode 打开项目后，AI 能在第一轮对话中正确回答"这个项目怎么跑测试""入口文件在哪""代码风格要求是什么"。

### Phase 2: 能力扩展（v1.5）
**目标**: 支持多种 IDE 和项目类型，扩大用户面。

| 任务 | 工作量 |
|------|--------|

**目标**: 建立 PBH 与生态工具的协作体系，形成"偏差检测→代价累积→归因锚定"闭环。

| 任务 | 工作量 |

| 实现 `harness-init validate` 命令 | 小 |

| **设计 PBH 与 Harness-Lint 的协作接口规范** | 中 |

| **建立三层防腐体系**（架构分层隔离 / CI 心智锚定 / 失败可见性） | 中 |

## 五、预期效果

### 分阶段预期效果

| 阶段 | 核心产出 | 生态意义 |
| :--- | :--- | :--- |
| v1.1.5（已完成） | 边界清晰、协议稳定、PyPI 可安装 | 奠定"环境播种"的基础能力 |
| v1.5.0 | 多项目骨架、IDE 适配、协议版本化 | 为生态工具提供稳定的依赖基础 |
| v2.0.0 | 生态接口、归因锚定、三层防腐体系 | PBH 从"工具"走向"基础设施" |

### 对用户
- 生成的项目干净，没有看不懂的 stubs
- AI 打开项目 30 秒内知道怎么工作
- 不管是做 CLI、Library、Web 还是 Notebook，都有对应起点
- 不用的 IDE 配置文件不会污染项目

### 对 PBH 项目
- 测试更稳定（砍掉依赖环境的集成测试）
- 定位清晰，叙事统一："AI 友好型项目脚手架"
- 与业界实践对齐：智谱"执行环境决定 Agent 能力"、OpenAI"Harness = 环境 + 约束 + 反馈"

### 对 README 的升级建议

在 README 增加"适用场景"段落，帮助用户第一眼建立正确心智：

```markdown
## 什么时候用 PBH？

- ✅ 你想快速启动一个结构标准化的 Python 项目
- ✅ 你需要 AI 编码助手能立即理解你的项目结构
- ✅ 你想用 Spec Coding 流程但不知道目录怎么组织
- ✅ 你需要在团队中统一项目结构和质量门禁

## 什么时候不用 PBH？

- ❌ 你需要一个 Agent 运行时框架 → 看 Hermes / CrewAI
- ❌ 你需要 Spec Coding 流程工具 → 自己写 PRD/SPEC
- ❌ 你已经有一套成熟的项目模板 → 继续用你的
```

**设计意图**：明确 PBH 的边界，不浪费用户时间，也不让 PBH 承担不该有的期待。
