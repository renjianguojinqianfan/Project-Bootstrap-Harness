# PBH 协议规范 (PBH-SPEC)

**版本**: v2.0.0-draft
**状态**: 草案，待定稿
**发布日期**: 2026-05-06
**本版要点**: 定义了标准化项目“协作宪法”与核心协议，标志着 PBH 从“工具”向“通用标准”的演进。

## 核心原则

1.  **协议即语言**：`AGENTS.md` 是 AI 与项目的共同“语言”。本规范旨在确保这种“语言”的一致性、完整性与清晰度。
2.  **约束即服务**：通过播种确定性的规则与门禁，为不确定的 AI 代理构建一个可预测、可验证的安全协作空间。
3.  **协议是代码**：一份足够详尽的协议，本身就是可以被准确执行、校验与复现的“代码”。

## 1. 问题域定义

当一个 AI 编码代理（Agent）进入一个新项目时，会面临三个核心信息缺口：

1.  **规则不可见**：项目特定的编码风格、测试要求、提交规范等规则，Agent 无法在首次接触时自动获取。
2.  **质量无从验证**：Agent 在生成代码后，没有一个统一的、自动化的入口来判定任务“是否完成”。
3.  **状态无从感知**：Agent 无法感知项目当前所处的开发阶段（初期/开发中/交付前），因而无法调整其工作策略。

PBH 的目标是在项目初始化时**系统性地解决**上述问题。它通过创建一套标准化的核心文件，为 AI 代理提供了一个**可预测、可验证、可跨语言复现**的工作环境。

## 2. 核心协议

### 2.1 `AGENTS.md` 规范

**目的**：为 AI 代理提供一份结构化的、可在 30 秒内解析的“项目操作手册”。

**实现要求**：任何兼容 PBH v2.0 的实现都**必须**能够生成或验证一个完全符合下述规范的 `AGENTS.md` 文件。

**格式要求**：
- **路径**：项目根目录下的 `AGENTS.md`。
- **格式**：符合 CommonMark 标准的 Markdown，带编号的章节标题。
- **Quick 变体**：兼容实现**可以**提供 `AGENTS.quick.md` 精简版（**必须** ≤ 50 行）。如果提供，其中必须包含 §2 Quick Start 和 §5 Critical Rules 章节的内容。

**强制章节定义**：
本规范明确区分了“必选 (MUST)”和“可选 (MAY)”的章节，实现者可根据此定义进行校验。

| 章节 | 必要性 | 内容要求 |
| :--- | :--- | :--- |
| §1 Project Snapshot | **必选** | 项目名称、一句话描述、项目类型（**必须**是 `cli`/`lib`/`web`/`notebook` 之一）及维护者。所有信息**必须**由实现从用户提供的元数据中渲染。 |
| §2 Quick Start | **必选** | 不超过 3 个步骤，供代理**验证环境是否正常并理解项目入口点**。第一步**必须**是 `make verify`。 |
| §5 Critical Rules | **必选** | 不可协商的硬性规则。**必须**包含：自我评价禁令（仅 `make verify` 输出为唯一依据）、安全禁令（禁止硬编码密钥）以及 `make verify` 提交门禁（绝不提交未通过门禁的代码）。 |
| §8 File Mapping | **必选** | 关键目录与文件的快速索引表。**必须**包含 `src/`、`tests/`、`docs/`、`tasks/`、`.harness/` 的条目。 |
| §9 Commands | **必选** | 常用开发命令的快速索引表。**必须**包含 `make verify`、`make test`、`make lint` 的条目。 |

**可选章节定义**：
| 章节 | 必要性 | 内容要求 |
| :--- | :--- | :--- |
| §3 Multi-Agent Notice | 可选 | 声明项目是否支持多代理协作，以及推荐的协作方式（如使用独立的 git worktree）。 |
| §4 Working Guidelines | 可选 | 代理应遵循的核心准则。所有约束**必须**以原则性表述呈现，而非数值限制。具体的数值限制**应当**放在 IDE 专属适配文件中。 |
| §6 / §7 | 预留 | 为未来扩展预留的章节编号。兼容实现**不得**使用这些编号。 |

### 2.2 `make verify` 规范

**目的**：为项目定义统一的质量门禁入口点。

**接口规范**：
- **命令**：`make verify`
- **行为预期**：**必须**在**一条命令**中完成项目所需的全部质量检查（如代码风格扫描、测试执行、覆盖率验证），确保项目处于“可交付”状态。
- **退出码约定**：
  - `0`：所有质量检查均已通过。
  - 非 `0`：至少一项检查失败。具体错误码可由各实现自定义，但**必须**与 CI/CD 工具的“失败”语义保持一致。

### 2.3 `.harness/progress.json` 规范

**目的**：为 AI 代理和上层编排器（如 Symphony、Multica）提供一个**结构化的、可查询的项目阶段快照**。

**实现要求**：任何兼容的实现都**必须**能够生成或校验符合下述数据结构的 `progress.json` 文件。

**Schema 定义**：
```json
{
  "project_name": "string",
  "current_stage": "enum['init','plan','execute','evaluate','done']",
  "plans": ["string"],
  "last_updated": "ISO 8601 datetime string"
}
```

| 字段 | 说明 |
| :--- | :--- |
| `project_name` | 项目名称，**必须**与 `AGENTS.md` 中声明的名称一致。 |
| `current_stage` | 项目当前阶段，**必须**是枚举值之一。详见下方生命周期定义。 |
| `plans` | 项目采用的外部规范或计划文件列表（如 `tasks/task_plan.md`），为 Agent 提供额外的上下文来源。 |
| `last_updated` | 最后更新的 ISO 8601 日期时间字符串。**必须**精确到秒。 |

**生命周期定义**：
| 阶段 | 含义 | Agent 行为预期 |
| :--- | :--- | :--- |
| `init` | 项目刚完成初始化。 | 先 `make verify` 确认环境干净后才可开始工作。 |
| `plan` | 正在进行架构设计与任务拆分。 | 只做规划与分析，不修改核心代码。 |
| `execute` | 正在实现功能。 | 仅激活质量规则（抑制风格噪声），以提升开发效率。 |
| `evaluate` | 准备交付。 | 严格检查所有规则，并对重复违规进行 CI 阻断。 |
| `done` | 当前版本已完成，项目处于稳定状态。 | 只做最小修复，不进行大改动。 |

## 3. 合规性定义

一个项目**可以**声明 **“符合 PBH v2.0 标准”** ，当且仅当其通过 `harness-init validate` 或等效工具的**自动化检查**：

1.  **文件存在性**：项目根目录下**必须**存在：
    *   一个符合 §2.1 格式规范的 `AGENTS.md` 文件。
    *   一个包含符合 §2.2 行为规范的 `verify` 目标的 `Makefile` 文件。
    *   一个符合 §2.3 Schema 规范的 `.harness/progress.json` 文件。
2.  **内容合规性**：上述文件的内容**必须**能通过任何实现了 §4.1 中定义的校验工具接口的自动化检查，且检查结果为“通过”。
3.  **行为合规性**：项目的 `make verify` 目标**必须**能成功运行，并返回正确的退出码。

## 4. 生态系统

### 4.1 配置与运行时

兼容实现**可以**提供一个可选的 `WORKFLOW.md` 文件支持。该文件应作为 PBH 生态系统中的一个可插拔组件，用于将项目规则与自动化流程或外部任务看板（如 Linear、GitHub Issues）集成。其格式、实现对 PBH 合规性声明**不构成**直接影响。

### 4.2 版本控制

- **协议版本**：本协议自身的版本（如 v2.0.0）独立于任何特定实现（如 `harness-init`）的版本。这是生态稳定性的基石。
- **兼容性机制**：`.harness/progress.json` **应当**包含一个 `harness_version` 字段，用于记录播种该项目时所采用的 PBH 协议版本。这对于确保不同版本的生态工具可以正确识别并处理项目至关重要。

## 5. 参考实现

官方维护的第一个参考实现是 Python 语言的 `harness-init`。它基于本规范（v2.0.0）的核心条款开发，是 PBH 生态的种子工具。我们鼓励社区根据本规范，贡献其他语言的实现或相关工具。
